2. Gitってなに？

ファイルのバージョン管理するソフトウェア。

  • ファイルの最新の状態がわかる

  • いつ・誰が・何を変更したかもわかる

  • 以前の状態に戻せる

  • 複数人での編集に対応している

2.1. Gitの歴史

リーナス・トーバルズがLinuxのカーネル開発で使用されていたバージョン管理システムのライセンス変更に伴い、次期バージョン管理システムを検討し始めたが、高度な要求水準を満たすものがなかった。

2005年からGitのプログラム開発が開始。

  • スピード

  • シンプルな設計

  • ブランチが並列で開発可能

  • 大規模プロジェクトが効率的に行える

2.2. GitHubってなに？

クリス・ワンストラスが創業者のサービス。コード（Gitリポジトリ）のホスティングサービス。

  • プルリクエストで複数人開発

    • 変更の取り込みをリクエスト、コミュニケーションが可能

    • コードレビューがしやすくなった

  • 世界中のチームがGitHub上で開発可能

    • 一般のエンジニアがオープンソース開発に参加可能

他にもBitbucketなどのホスティングサービスもある。 <<< == Git環境構築 :chapter: 3

2.3. Gitのインストール

  1. 下記のサイトからGit for Windowsのインストーラをダウンロードする。

    Git for Windows

  2. 全部初期設定のまま、インストールする。

  3. インストールした中にある「Git Bash」を起動し、下記のコマンドを投入する。

    git -version

  4. インストールしたバージョンが表示されることを確認する。

2.4. Githubへの登録

  1. 下記のサイトからGithubへサインアップする。

    Github

  2. Email認証を行う。

  3. Githubにログインできることを確認する。

2.5. Gitの初期設定

  1. 「Git Bash」を起動する。

  2. 下記のコマンドを投入し、ユーザ名を登録する。

    git config --global user.name "hogehoge"

  3. 下記のコマンドを投入し、メールアドレスを登録する。

    git config --global user.email hoge@hoge.com

  4. 下記のコマンドを投入し、エディターを設定する。

    Visual Studio Code
    git config --global core.editor 'code --wait'
    Atom
    git config --global core.editor 'atom --wait'

  5. 下記のコマンドを投入し、設定を確認する。

    git config user.name // (1)
git config user.email // (2)
git config core.editor // (3)
git config --list //(4)

    1. ユーザ名表示

    2. メールアドレス表示

    3. エディター表示

    4. 全設定をリスト表示

ここで設定した設定は「C:\\Users\username\.gitconfig」に保存されている。

3. Gitの仕組み

3.1. Gitのメリット

Q.Gitのバージョン管理ではスナップショットで保存しているか？それとも差分で保存しているか？

A.スナップショット

まるまるデータを保存していくため、ブランチを切ったり、マージしたりがしやすくなった。

コミットをたどることで以前の状態に辿りながら戻せる。

3.2. Gitの作業の流れ

  1. ワークツリー領域でファイルを作成

  2. 「git add」コマンドを実行

  3. ローカルリポジトリ領域にファイル内容を元にした圧縮ファイルを作成

  4. ステージ領域に圧縮ファイルとファイル名を関連付けるインデックスを作成

  5. 「git commit」コマンドを実行

  6. ローカルリポジトリ領域にステージ領域のインデックスを元にツリーを作成

  7. ローカルリポジトリ領域にツリーのプロパティ（親コミット、作成者、日付、コメント）を示すコミットを作成

3.3. Gitのデータ構造

  • ローカルリポジトリに「圧縮ファイル」「ツリー」「コミット」ファイルを作成することでデータを保存している

  • コミットが親コミットを持つことで変更履歴を辿る事ができる

  • Gitとは「データを圧縮して、スナップショットとして保存、ツリーとコミットによって、時系列の依存関係を保存し、バージョン管理をしているアプリケーション」 <<< == Gitの基本 :chapter: 4

3.4. 新規でローカルリポジトリを作ろう

  1. 「Git Bash」を起動する

  2. プロジェクトのディレクトリを作成し、直下に移動する

    mkdir [ディレクトリ名]
cd [ディレクトリ名]

  3. 下記のコマンドを投入し、ローカルリポジトリを作成する

    git init

3.5. リモートリポジトリを元にローカルリポジトリを作ろう

  1. リモートリポジトリのURLを入手する

  2. 「Git Bash」を起動する

  3. プロジェクトのディレクトリを作成したいディレクトリの直下に移動する

    cd [ディレクトリ名]

  4. 下記のコマンドを投入し、リモートリポジトリを複製する

    git clone [リモートリポジトリのURL]

3.6. ファイルの変更をステージに追加しよう

Q . なぜステージ領域を使いますか？

A . リポジトリへ変更したい変更だけを選択するため。

  1. 下記のコマンドを投入し、ファイルの変更をステージに追加する

    変更範囲を指定
    git add [ファイル名またはディレクトリ名]
    全範囲を指定
    git add .

3.7. ファイルの変更を記録しよう

  1. 下記のコマンドを投入し、ファイルの変更を記録する

    optionなし
    git commit

  2. エディターが立ち上がる

  3. エディターにコミットメッセージを記載し、保存する

  4. ターミナルを確認し、コミットメッセージ、コミットした内容が表示されていることを確認する

3.8. どのファイルが変更されたかを確認しよう

  1. 下記のコマンドを投入し、ファイルの変更を確認する

    git status

  2. ターミナルを確認し、ワークツリーとステージ、ステージとコミットの変更点を確認する

3.9. ファイルの変更内容を確認しよう

  1. 下記のコマンドを投入し、ファイルの変更を確認する

    ワークツリーとステージ
    git diff [ファイル名]
    ステージとコミット
    git diff --staged [ファイル名]

  2. ターミナルを確認し、ワークツリーとステージ、ステージとコミットの変更点を確認する

3.10. ファイルを削除しよう

  1. 下記のコマンドを投入し、ファイルを削除する

    リポジトリとワークツリーから削除する
    git rm [ファイル名]
git rm [ディレクトリ名]
    リポジトリからのみ削除する
    git rm --cached [ファイル名]

  2. 「ls」または「git status」で削除されていることを確認する

3.11. ファイルを移動（またはリネーム）しよう

  1. 下記のコマンドを投入し、ファイルを移動する

    git mv [旧ファイル名] [新ファイル名]

    下記を順に実行するのと同じ。

    mv [旧ファイル名] [新ファイル名]
git rm [旧ファイル名]
git add [新ファイル名]

  2. 「ls」または「git status」で移動されていることを確認する

3.12. ファイルをリモートリポジトリに記録しよう

  1. リモートリポジトリを作成する

    Githubの場合

    1. Githubにログインする

    2. 右上のプロフィールメニューから「Your Repository」をクリックする

    3. 表示された画面の「Repository」タブをクリックする

    4. NEWをクリックする

    5. リポジトリ名を入力し、Create Repositoryをクリックする

    6. リポジトリが作成されたことを確認する

  2. 下記のコマンドを投入し、リモートリポジトリを設定する

    git remote add [リモート名] [リモートリポジトリのアドレス]

  3. 下記のコマンドを投入し、ファイルをリモートリポジトリに記録する

    git push [リモート名] [ブランチ名]

  4. リモートリポジトリが更新されていることを確認する

3.13. コマンドにエイリアスをつけよう

  1. 下記のコマンドを投入し、エイリアスを設定する

    git config --global alias.[エイリアス名] [コマンド名]
    git config --global alias.ci commit
git config --global alias.st status
git config --global alias.br branch
git config --global alias.co checkout

3.14. バージョン管理しないファイルを無視しよう

Q . どんなファイルはバージョン管理しない？

A . 機密情報やチーム開発に利用しないファイル
.gitignoreファイルを利用する

  1. 「.gitignore」ファイルを追加し、中身に無視したいファイル名やディレクトリ名を記載する

3.15. ワークツリーのファイルの変更を元に戻そう

  1. 下記のコマンドを投入し、ファイルをステージの状態に戻す

    変更範囲を指定
    git checkout -- [ファイル名またはディレクトリ名]
    全範囲を指定
    git checkout -- .

  2. ワークツリーの内容が戻っていることを確認する

3.16. ステージのファイルの変更を元に戻そう

  1. 下記のコマンドを投入し、ファイルをステージの状態に戻す

    変更範囲を指定
    git reset HEAD [ファイル名またはディレクトリ名]
    全範囲を指定
    git reset HEAD .

  2. ステージの内容が戻っていることを確認する

注記

このコマンドでワークツリーへは何も変化が起きないことに注意。

3.17. 直前のコミットをやり直そう

  1. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    git commit --amend

  2. コミットの内容が戻っていることを確認する

警告

このコマンドはリモートリポジトリにプッシュしたコミットは修正しないこと。プッシュした内容を修正したい場合は、追加で過去のコミットは取り消さず、修正したものを改めてコミットをする。

3.18. リモートリポジトリの情報を確認しよう

  1. 下記のコマンドを投入し、リモートリポジトリの情報を確認する

    リモート名表示
    git remote
    リモート名とURLを表示
    git remote -v

  2. リモートリポジトリの内容を確認する

3.19. リモートリポジトリを登録しよう

  1. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    リモート名表示
    git remote add <リモート名> <リモートURL>

3.20. リモートリポジトリの変更を取得しよう1

  1. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    リモート名表示
    git fetch <リモート名>

  2. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    リモート名表示
    git merge <リモート名>/<ブランチ名>

  3. マージメッセージ

3.21. リモートリポジトリの変更を取得しよう2

  1. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    リモート名表示
    git pull <リモート名> <ブランチ名>

  2. マージメッセージ

警告

pullは現在のブランチにmergeする。
自分のカレントブランチを意識しないと意図しないマージをしてしまうので、注意。

3.22. リモートリポジトリの詳細情報を確認しよう

  1. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    リモート名表示
    git remote show <リモート名>

  2. マージメッセージ

3.23. リモートリポジトリの情報を編集しよう

  1. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    git remote rename <旧リモート名>　<新リモート名>
    git remote rm <リモート名>

  2. マージメッセージ

4. ブランチの仕組み

ブランチ とは、並行して複数機能を開発するため にある仕組み。

Q.ブランチのメリットは？*

A.ブランチを分岐させることで、自分の変更内容を周りに影響させず、逆に回りの変更内容を自分の変更に影響させずに開発をすすめることができる

4.1. ブランチの考え方

ブランチはあくまで コミット(コミットID)を指し示すポインタ。コミットすると基本的には自分が作業しているブランチ（=HEAD）の指し示すコミットが新しいコミットに移動する。
HEADは ブランチをを指し示すポインタ。 ブランチの情報は「.git」ディレクトリ配下に保存されている。

Gitにおいて、ブランチの作成や切り替えが単なるポインタの切り替えで済むため、非常に高速に処理できるのが、他のバージョン管理システムとの大きな特徴。 <<< == Gitでブランチを利用する

4.2. ブランチの情報を確認しよう&新規でブランチを作成しよう

git branch
git merge <リモート名/ブランチ名>

  • Auto Merge
    基本的なマージ。HEADのブランチをベースにマージコミットという新しいコミットファイルを作成し、そのコミットファイルをポインタとする。
    マージコミットの親コミットは2つとなる。

  • Fast Foward
    ブランチが枝分かれしていなかったときのマージ。単にHEADのブランチのポインタを前にすすめるだけ。

  • Conflict
    ブランチ間で変更内容の競合をが起きている場合、両方の変更が反映される

4.3. 他のブランチの内容をマージしよう

  1. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    git merge <ブランチ名>

4.4. コンフリクトを起こさないための運用・心構え

  • 複数人で同じファイルを変更しない

  • pullやmergeする前に変更中の状態をなくしておく（commitやstashしておく）

  • pullするときは、pullするブランチに移動してからpullする

  • コンフリクトしても慌てない

4.5. ブランチを編集・削除しよう

  1. 下記のコマンドを投入し、コミットをひとつ前の状態に戻す

    ブランチ名の変更
    git branch -m <新ブランチ名>
    削除
    git branch -d <ブランチ名>
    強制削除
    git branch -D <ブランチ名>

4.6. 開発とブランチの使い方

  • masterブランチをリリース用ブランチに、開発はトピックブランチを作成してすすめるのが基本

4.7. リモートブランチとは

  • リモートブランチとはリモートリポジトリにあるブランチへのポインタ　。 <リモート名><ブランチ名>で参照できる <<< == Githubを利用した開発
